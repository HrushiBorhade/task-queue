# Task Queue — Project Context

Distributed async task orchestration platform built as a Bun monorepo.

## Architecture

- **apps/web** — Next.js 16.1.6 frontend + API routes (Vercel deployment)
- **apps/worker** — BullMQ worker process (7 task workers + scheduler)
- **apps/trigger** — Trigger.dev integration (alternative orchestration)
- **packages/shared** — Shared types, queue config, Drizzle ORM schema

## Tech Stack

| Layer | Technology |
|-------|-----------|
| Runtime | Bun (monorepo workspaces) |
| Frontend | Next.js 16, React 19, Tailwind v4, shadcn/ui, Motion |
| State | TanStack Query v5 (server state, optimistic updates, infinite scroll) |
| Auth | Supabase Auth (Google OAuth + email/password, PKCE flow) |
| Database | Supabase Postgres + Drizzle ORM, RLS on all tables |
| Queue | BullMQ + Redis 7 (per-queue concurrency, rate limits, retries) |
| Realtime | Supabase Realtime (postgres_changes + broadcast channels) |
| Observability | Sentry, Prometheus (prom-client), Pino, PostHog, Vercel Analytics, Clarity |
| Testing | Vitest, Playwright, Testing Library |
| Icons | Phosphor Icons (weight="duotone") |
| Animations | motion/react (LazyMotion + m for tree-shaking) |

## Key Patterns

### Database Workflow
1. `bun db:new <name>` → write SQL → `bun db:push` → `bun db:pull:types` → update `schema.ts`
2. Frontend types: `apps/web/lib/database.types.ts` (generated by Supabase CLI)
3. Worker schema: `packages/shared/src/schema.ts` (manual Drizzle schema, mirrors DB)

### Auth — Three Supabase Clients
- **Browser** (`lib/supabase/client.ts`) — publishable key, client-side auth + realtime
- **Server** (`lib/supabase/server.ts`) — publishable key + cookies, SSR auth checks
- **Admin** (`lib/supabase/admin.ts`) — service role key, bypasses RLS (worker + admin routes)

### Worker Architecture
- Single Bun process runs 7 BullMQ workers + scheduler
- Health check on `:9090`, Prometheus metrics on `:9092`, Bull Board on `:9091`
- Direct Postgres via Drizzle (not Supabase client) — bypasses RLS
- Graceful shutdown: waits for in-flight jobs on SIGINT/SIGTERM

### Task Types (7)
`text_gen`, `image_gen`, `research_agent`, `email_campaign`, `pdf_report`, `webhook_processing`, `data_aggregation`

### Shared Utilities
- `lib/task-utils.ts` — `timeAgo()`, `formatLabel()`, `getImageOutput()`, `TYPE_ICON`, `TASK_STATUS_BADGE`
- `lib/query-keys.ts` — TanStack Query key factory
- `lib/analytics.ts` — PostHog event tracking wrapper

## Commands

```bash
bun run dev          # Start all apps
bun run dev:web      # Next.js only
bun run dev:worker   # Worker only
bun run db:new <n>   # New migration
bun run db:push      # Apply migrations
bun run db:reset     # Reset local DB
```

## Conventions

- Use Bun instead of Node.js/npm/pnpm for all commands
- Bun auto-loads `.env` — no dotenv needed
- `bun install` from monorepo root (Bun downloads once, workspace node_modules are symlinks)
- shadcn/ui components: use `FieldGroup` + `Field` for forms, `data-icon` on icons in buttons
- Animations: use `LazyMotion` + `m` from `motion/react` (not bare `motion`) for bundle savings
- All tables have RLS — API routes check auth independently of middleware
- Security headers configured in `next.config.ts` (HSTS, CSP, X-Frame-Options DENY)
- Sentry tunnel via `/monitoring` route to avoid ad blockers
